# MVP 范围清单 V1

## 1. 文档目标

这份文档用于把第一版 MVP 的范围明确成一份可检查、可验收的清单。

它回答的问题是：

- 第一版必须做什么
- 第一版明确不做什么
- 什么标准算“已经达到可演示的 MVP”
- 哪些能力属于后续版本

这份文档的目标是帮助我们在真正开始实现时，控制范围、减少膨胀、提升交付确定性。

## 2. MVP 总目标

第一版 MVP 的目标不是做一个功能很多的量化平台，而是做一个：

- 基于 Google ADK
- 能把自然语言问题转成结构化策略
- 能进行可解释回测
- 能输出收益曲线和核心指标

的研究型产品原型。

一句话说：

第一版要证明“自然语言量化研究”这条主链路是可以跑通的。

## 3. MVP 核心主链路

第一版必须打通下面这条链路：

1. 用户输入自然语言问题
2. 系统理解并生成策略草案
3. 系统识别缺失字段并做澄清
4. 系统生成 Strategy Schema
5. 系统执行回测
6. 系统计算核心指标
7. 系统输出结果解释
8. 系统展示回测收益曲线

如果这条链路跑不通，第一版就还不算完成。

## 4. 第一版必须支持的用户场景

## 4.1 场景 A：ETF 技术指标策略

必须支持的典型问题：

- “对于沪深300ETF，MACD 日线金叉买入、死叉卖出，每年的平均收益是多少？”

### 必须支持的能力

- ETF 标的识别
- 单标的信号策略解析
- MACD / 均线类基础指标
- 回测收益曲线
- 年化收益、最大回撤、夏普
- 基础结果解释

## 4.2 场景 B：A 股横截面轮动策略

必须支持的典型问题：

- “如果每个月买入市值最大的 20 只股票，持有到下个月，收益是多少？”

### 必须支持的能力

- A 股股票池默认识别
- `daily_basic` 市值字段读取
- Top N 轮动逻辑
- 月度调仓
- 等权组合
- 组合收益曲线
- 年度收益表
- 换手率等基础统计

## 7. 第一版明确不纳入的能力

这部分非常重要，必须显式写出来。

## 7.1 市场与资产范围

- [ ] 不支持期货
- [ ] 不支持期权
- [ ] 不支持可转债
- [ ] 不支持分钟级与 Tick 级回测

## 7.2 交易能力

- [ ] 不支持实盘自动下单
- [ ] 不支持杠杆
- [ ] 不支持做空
- [ ] 不支持复杂订单类型

## 7.3 Agent 能力

- [ ] 不做复杂网状多 Agent 协商
- [ ] 不做重型长期记忆系统
- [ ] 不做大规模并行参数搜索

## 7.4 数据与运维

- [ ] 第一版不把 Tushare 在线拉数放入主回测路径
- [ ] 第一版不要求迁移完整的定时数据维护体系
- [ ] 第一版不要求完整 MCP 数据源生态

## 7.5 展示与分析

- [ ] 不要求行业暴露分析
- [ ] 不要求月度热力图
- [ ] 不要求复杂因子归因
- [ ] 不要求高级可视化工作台

## 9. 第一版演示脚本建议

为了方便后续演示或自测，建议准备一组固定演示问题。

### 演示问题 1

## 10. MVP 判断标准

如果下面这几项都成立，就可以认为第一版 MVP 已经成立：

- [ ] 能稳定跑通两类核心策略场景
- [ ] 能通过自然语言收敛成 Strategy Schema
- [ ] 能输出回测收益曲线
- [ ] 能输出关键收益与风险指标
- [ ] 能给出基本解释和风险提示
- [ ] 能在 ADK Session / State / Artifact 中沉淀研究结果

## 11. 第一版最容易失控的地方

为了帮助后面控制范围，这里也明确列出来。

### 风险点 1：过早做太多策略类型

建议：

- 先打透 ETF 技术策略和横截面轮动

### 风险点 2：过早追求复杂 UI

建议：

- 第一版先把收益曲线和关键指标做对

### 风险点 3：过早迁移完整数据维护系统

建议：

- 第一版先使用当前本地已有数据

### 风险点 4：把 ADK 用得过重

建议：

- 第一版先做清晰主链路，不做复杂多 Agent 网络
