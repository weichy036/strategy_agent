# Google ADK 架构原则

## 1. 总原则

当前项目必须基于 Google ADK 来设计整个 Agent 系统。

这不是一句“以后会接入 ADK”的口号，而是一个明确的架构约束：

- Agent 编排基于 ADK
- 对话上下文基于 ADK
- 工具调用基于 ADK
- 会话状态管理尽量贴合 ADK
- 多阶段任务流转围绕 ADK 的抽象组织

换句话说，我们要做的是一个 ADK Native 的量化研究产品。

## 2. 为什么必须是 ADK Native

如果先自己搭一套独立的 Agent 编排框架，再在外层包一层 ADK，会有几个问题：

- 概念重复，维护成本高
- 运行链路不清晰
- 工具协议容易分叉
- 会话、状态、记忆层容易出现两套体系

所以更合理的方式是：

- 尽早把产品需求映射到 ADK 的核心抽象
- 让 ADK 成为系统中枢，而不是附属层

## 3. 当前可对齐的 ADK 抽象

从已安装包的接口来看，当前项目后续最值得围绕的对象包括：

- `Agent`
- `Runner`
- `tools`
- `sessions`
- `memory`
- `artifacts`
- `flows`

这几个对象足以支撑我们第一版系统建模。

## 4. 我们的业务对象如何映射到 ADK

下面是一个建议的一一映射关系。

### 用户研究会话

业务含义：

- 用户围绕某个投资想法展开的一段连续研究过程

建议映射：

- ADK `Session`

### 对话研究助手

业务含义：

- 理解用户问题
- 补全歧义
- 组织后续动作

建议映射：

- 顶层 `Agent`

### 策略解析器

业务含义：

- 把自然语言问题转换成结构化策略草案

建议映射：

- 子 `Agent` 或专用 `Tool`

取决于后续我们希望它更偏“推理”还是更偏“规则转换”。

### 回测执行器

业务含义：

- 接收策略对象
- 读取数据
- 运行回测
- 返回结构化结果

建议映射：

- `Tool`

因为它本质上是一个确定性执行模块。

### 结果解释器

业务含义：

- 把收益、风险、稳定性等结果翻译成用户可理解的语言

建议映射：

- 子 `Agent`

因为它更偏分析和表达，而不是单纯计算。

### 策略卡片 / 报告 / 图表

业务含义：

- 用户研究过程中的结构化输出

建议映射：

- `artifacts`

## 5. 一个推荐的第一版 ADK 分层

### 顶层：Research Orchestrator Agent

职责：

- 接收用户问题
- 判断问题类型
- 决定是否需要澄清
- 调用解析、回测、解释能力
- 组织最终回复

### 第二层：领域子 Agent

建议先有两个：

- Strategy Parsing Agent
- Result Explanation Agent

### 第三层：确定性工具层

建议先有以下工具：

- Instrument Resolve Tool
- Strategy Validate Tool
- Market Data Query Tool
- Backtest Run Tool
- Metrics Compute Tool

### 第四层：状态与产物层

用于保存：

- 会话上下文
- 策略对象
- 回测结果
- 图表或报告

## 6. 哪些能力不应该做成 Agent

为了避免“万物皆 Agent”，有些模块应该优先做成 Tool 或普通库：

- 指标计算
- 数据读取
- 回测执行
- 指标统计
- Schema 校验

这些能力的特点是：

- 输入输出清晰
- 逻辑应确定
- 不需要开放式推理

因此更适合作为 Tool 或底层模块。

## 7. 第一版架构边界

第一版建议坚持：

- 一个顶层编排 Agent
- 少量子 Agent
- 多个确定性 Tool

不建议一开始就做非常复杂的多 Agent 网状协作，因为当前最重要的是把研究主链路跑通：

自然语言问题 -> 策略结构化 -> 回测执行 -> 结果解释

## 8. 文档和设计上的落地要求

从现在开始，后续每一份系统设计文档都应该明确回答：

1. 这个模块在业务上负责什么
2. 它在 ADK 中对应什么抽象
3. 它与其他 Agent / Tool / Session / Artifact 如何协作

如果一个设计无法回答这三个问题，说明它还没有真正对齐到 ADK。

## 10. 结论

这个项目后面不是“使用 Python 做一个回测系统，再接一点 Agent 能力”。

而是：

- 用 Google ADK 作为系统编排底座
- 用确定性数据和回测模块作为工具层
- 用自然语言交互把量化研究能力产品化

这条路线能最大程度保证后续系统既有 Agent 的表达力，又不会丢掉金融场景所需要的确定性和可信度。
