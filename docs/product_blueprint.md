# 产品蓝图

## 1. 为什么要做这个产品

今天大模型已经极大降低了“写代码”的门槛，但在量化投资领域，真正的门槛并不只是代码本身，而是以下几个环节：

- 用户不知道如何把一个投资想法表达成严谨的策略定义
- 用户不知道回测需要哪些假设和交易约束
- 用户不知道如何判断结果是否可信
- 用户不知道收益是来自真实逻辑，还是来自数据偏差和过拟合

因此，这个产品的真正价值不是“帮用户生成一段策略代码”，而是帮助用户完成一整套研究闭环：

提出假设 -> 明确规则 -> 回测验证 -> 风险解释 -> 保存复用

## 2. 产品使命

让普通投资者也能用自然语言完成结构化策略研究，并清楚理解：

- 自己买的是什么
- 为什么买
- 在什么条件下买卖
- 历史上结果如何
- 风险来自哪里
- 这个结果为什么可能不可靠

## 3. 目标用户

### 第一类：普通投资者

特点：

- 有投资兴趣，但缺少量化、编程和统计背景
- 常通过新闻、博主、朋友推荐形成交易想法
- 想验证一个直觉是否值得相信

核心需求：

- 把一句人话变成回测
- 看懂策略到底在干什么
- 避免“听消息买入”式决策

### 第二类：半专业研究者

特点：

- 懂一些指标、选股逻辑和回测概念
- 可能会写一点 Python，但效率不高
- 希望快速验证大量假设

核心需求：

- 快速搭建策略原型
- 调参和比较多个策略版本
- 输出结构化报告

### 第三类：投顾 / 内容创作者 / 教学用户

特点：

- 需要把策略讲给别人听
- 需要展示“逻辑、证据、风险”而不是单纯结论

核心需求：

- 生成可展示的策略报告
- 让策略解释更清楚、更有教育性

## 4. 核心产品定位

### 我们做什么

- 自然语言策略研究
- 规则型量化回测
- 策略解释与风险揭示
- 策略模板保存与复用

### 我们暂时不做什么

- 实盘自动交易
- 高频交易
- 复杂衍生品定价
- 黑盒荐股
- 以“预测涨跌”为核心卖点的产品

这个边界非常重要。因为一旦产品从“研究辅助”滑向“直接给结论”，它在产品可信度、监管风险和用户预期管理上都会迅速变难。

## 5. 核心价值主张

对用户来说，最重要的不是“代码自动生成”，而是下面四件事：

### 价值 1：把模糊想法变成明确规则

例如用户说：

“MACD 金叉买入，死叉卖出。”

系统需要自动补全并确认：

- 标的是什么
- 周期是日线还是小时线
- 是否允许重复开仓
- 买入后是否满仓
- 是否考虑手续费和滑点
- 回测区间是什么

### 价值 2：把结果解释清楚

回测不能只输出“年化 18%”，还要解释：

- 盈利主要来自哪一段行情
- 最大回撤出现在什么阶段
- 交易频率是否过高
- 对交易成本是否敏感
- 策略在不同年份是否稳定

### 价值 3：让研究过程可复现

每一次分析都应当能沉淀为一个可复用对象：

- 原始问题
- 结构化策略定义
- 参数
- 数据版本
- 回测结果
- 图表与解释

### 价值 4：帮助用户避免常见认知陷阱

系统应主动提示：

- 这是不是未来函数
- 是否存在幸存者偏差
- 是否忽略停牌涨跌停和流动性
- 是否样本太短
- 是否过拟合

## 6. 典型用户场景

### 场景 A：单标的技术策略

用户问题：

“对于沪深 300 ETF，MACD 日线金叉买入、死叉卖出，每年的平均收益是多少？”

系统流程：

1. 识别标的：沪深 300 ETF
2. 识别策略：MACD 金叉 / 死叉
3. 识别频率：日线
4. 自动补全默认参数：12, 26, 9
5. 自动补全执行规则：下一交易日开盘成交
6. 加入基础手续费和滑点
7. 输出年化、逐年收益、回撤、交易明细与解释

### 场景 B：横截面选股策略

用户问题：

“如果每个月买入市值最大的 20 只股票，持有到下个月，收益是多少？”

系统流程：

1. 识别股票池：A 股全市场
2. 识别选股因子：总市值从大到小
3. 识别持仓数：20 只
4. 识别调仓频率：每月
5. 自动补全权重规则：等权
6. 自动补全过滤项：剔除停牌、ST、新股、退市风险
7. 输出组合收益、换手率、分年度表现、行业暴露和解释

### 场景 C：组合条件筛选

用户问题：

“选出过去 60 天创新高、且 20 日均线向上、且换手率高于过去一年中位数的股票，每周五收盘买入，下周五卖出。”

系统流程：

1. 将自然语言解析为多条件筛选表达式
2. 明确买卖时点和可交易价格
3. 运行事件驱动或规则驱动回测
4. 提供分组统计与稳定性分析

## 7. 产品形态设计

我建议产品采用“三层界面”。

### 第一层：对话式入口

用户用自然语言提出问题。

系统返回：

- 我理解你的策略是……
- 还缺少这些关键假设……
- 我将按这些默认值执行……

这一层负责降低门槛。

### 第二层：结构化策略卡片

系统把自然语言转换为一个可见的结构化策略定义卡：

- 标的 / 股票池
- 买入条件
- 卖出条件
- 调仓频率
- 仓位规则
- 手续费 / 滑点
- 回测区间
- 过滤条件

这一层负责建立信任。

### 第三层：研究结果页

建议包含：

- 累计收益曲线
- 回撤曲线
- 年度收益表
- 风险指标卡片
- 持仓与换手统计
- 交易记录
- 系统解释
- 局限性与风险提示

这一层负责帮助用户真正理解结果。

## 8. 核心产品能力

### 8.1 自然语言解析

把用户问题解析为结构化策略定义。至少要识别：

- 标的 / 股票池
- 指标与参数
- 条件表达式
- 调仓频率
- 排序规则
- 持仓数量
- 权重规则
- 买卖价格假设
- 回测区间
- 成本设置

### 8.2 歧义澄清

很多自然语言问题并不完整，系统必须自动识别缺失信息，并使用以下策略：

- 优先从语义中推断
- 推断不了时提供默认值
- 对关键影响项要求用户确认

关键影响项包括：

- 股票池范围
- 调仓频率
- 买卖价格
- 是否复权
- 是否考虑成本
- 过滤规则

### 8.3 策略中间表示

自然语言不能直接驱动回测，必须先转成统一的策略表示。

建议采用 JSON Schema 或 DSL 表示，作为系统核心中枢。这样有三个好处：

- LLM 只负责“理解和生成结构”
- 回测引擎只负责“执行结构”
- 结果解释层只负责“解释结构和结果”

### 8.4 回测执行引擎

第一阶段建议支持：

- 单标的择时
- 多标的定期调仓
- 条件选股
- 排序选股
- 固定持有期策略

回测要至少处理：

- 交易日历
- 复权价格
- 手续费和滑点
- 停牌
- 涨跌停无法成交
- 仓位分配
- 调仓执行顺序

### 8.5 结果解释引擎

输出不能停留在数字层面，应该生成可读解释：

- 该策略在震荡市表现较差，主要因频繁假信号导致交易成本累积
- 收益主要集中在某几年，稳定性一般
- 高换手导致净收益对滑点较敏感
- 该策略偏向大盘价值风格

### 8.6 对比与复用

用户应能：

- 保存策略
- 克隆策略
- 改参数重新跑
- 并排比较两个策略
- 保存为模板

## 10. 技术架构建议

建议采用六层架构。

并且这里要补充一个非常明确的工程约束：

- 整个系统必须基于 Google ADK 这个 Agent 框架来设计
- 我们不是先做一套普通 Python 服务，再“接一个 ADK 外壳”
- 而是从对话编排、上下文管理、工具调用、任务流转开始，就按 ADK Native 思路建模

### 10.1 应用层

负责：

- Web / App 对话入口
- 策略卡片编辑
- 回测结果展示
- 策略保存与比较

### 10.2 Agent 编排层

负责：

- 理解用户问题
- 提取策略要素
- 判断歧义
- 发起澄清问题
- 组装结构化策略
- 调用回测和解释模块

这一层后续应直接映射到 Google ADK 中的 Agent、Runner、Tool、Session、Memory 等抽象，而不是自定义一套平行编排框架。

### 10.3 策略表示层

负责：

- 统一的 Strategy Schema
- 参数合法性校验
- 版本管理
- 默认值补全

这是整个系统最关键的一层。

### 10.4 数据与因子层

负责：

- 行情数据
- 复权处理
- 基础财务字段
- 指标计算
- 股票池过滤
- 行业、市值、风格标签

### 10.5 回测引擎层

负责：

- 订单撮合简化
- 持仓和现金账户
- 调仓逻辑
- 成本与约束
- 指标计算

### 10.6 解释与报告层

负责：

- 绩效摘要
- 风险归因
- 可读说明
- 风险提示
- 可分享报告

## 11. 一条完整执行链路

一条用户问题的执行路径建议是：

1. 用户输入自然语言问题
2. LLM 提取结构化草案
3. Schema 校验器检查字段完整性
4. 缺失关键字段时发起澄清
5. 生成最终策略对象
6. 数据层准备所需数据
7. 回测引擎执行
8. 分析模块计算指标
9. 解释模块生成结论和局限性
10. 前端返回策略卡 + 图表 + 解释

## 12. MVP 建议范围

如果我们想把第一版真正做出来，建议严格收敛到下面这些能力。

### 支持的市场

- A 股
- ETF

### 支持的策略类型

- 单标的技术指标择时
- 月 / 周频定期调仓
- 横截面排序选股
- 固定持有期事件策略

### 支持的输入能力

- 指标类：MA、EMA、MACD、RSI、布林带
- 价格类：突破新高、新低、涨跌幅
- 横截面类：市值、成交额、波动率、动量
- 条件组合：and / or / not

### 支持的输出能力

- 累计收益
- 年化收益
- 最大回撤
- 夏普比率
- 胜率
- 换手率
- 年度收益表
- 交易记录

### 暂不支持

- 期货、期权、可转债
- 分钟级和 Tick 级回测
- 复杂强化学习策略
- 自动下单

## 13. 最关键的产品难点

### 难点 1：自然语言歧义

用户说“买入市值最大的 20 只股票”，至少还缺少：

- 哪个市场
- 哪个时间点的市值
- 什么时候调仓
- 等权还是市值加权
- 是否过滤 ST、新股、停牌

解决思路：

- 建立“必填字段 + 默认规则 + 澄清机制”

### 难点 2：回测可信度

如果没有明确处理：

- 幸存者偏差
- 未来函数
- 成本
- 不可交易约束

那回测结果会严重失真。

解决思路：

- 把可信度约束做成系统默认机制，而不是用户可选项

### 难点 3：解释能力

多数用户并不缺“一个收益数字”，而是缺“这个数字意味着什么”。

解决思路：

- 结果页必须自带解释和风险揭示，而不是只给图

### 难点 4：表达能力和可控性平衡

如果语言过于自由，系统很难稳定执行；如果过于结构化，用户门槛又变高。

解决思路：

- 前端用自然语言
- 中间层用结构化 Schema
- 必要时允许用户在策略卡片中做显式编辑

## 14. 产品护城河应该是什么

我认为这个产品真正的护城河不会是“能不能写出回测代码”，而会是以下几个东西：

- 高质量的策略中间表示设计
- 面向金融研究的歧义澄清系统
- 一套可信的默认回测约束
- 强解释能力
- 用户策略资产的沉淀和复用

换句话说，护城河是“研究闭环和可信度体系”，不是“单次问答能力”。

## 15. 商业化方向

可以考虑三层收费。

### 免费版

- 每月有限回测次数
- 基础策略模板
- 基础结果展示

### 专业版

- 更多回测次数
- 更长历史区间
- 更多指标和策略类型

### 高级版 / B 端

- 团队协作
- 策略知识库
- API 接入
- 合规模板
- 私有数据源

## 16. 一个合理的落地节奏

### Phase 1：研究型 MVP

目标：

- 跑通“自然语言 -> 结构化策略 -> 回测 -> 解释”

重点：

- 只支持 A 股 / ETF
- 只支持日频
- 只支持 3 到 5 类最常见策略

### Phase 2：策略工作台

目标：

- 支持保存、克隆、比较、参数扫描

重点：

- 策略版本化
- 多策略对比
- 模板市场

### Phase 3：智能研究助手

目标：

- 让系统不仅执行，还能提出改进建议

例如：

- “这个策略对 2020 年行情依赖较强”
- “把调仓频率从周度改为月度后，换手下降且收益基本持平”
- “加入流动性过滤后结果更稳健”

## 17. 我对第一版产品的建议结论

如果我们想做一个真正能打的产品，第一版一定要坚持下面这条路线：

- 不是做一个会说金融术语的聊天机器人
- 不是做一个简单包装的 backtest notebook
- 而是做一个“自然语言驱动的、基于 Google ADK 的、可解释的策略研究操作系统”

最先要打透的，不是策略数量，而是这一条主链路：

用户问题 -> 策略结构化 -> 可可信回测 -> 可解释结果

只要这条链路跑顺，后面无论接更多指标、更多市场、更多策略类型，都会变成扩展问题，而不是重做问题。

## 18. 我建议我们下一步继续设计的三个对象

接下来最值得继续往下拆的是：

1. 第一版 Strategy Schema
2. 第一版系统架构与服务拆分
3. 第一版对话交互流程和澄清机制

如果继续推进，我建议下一轮我们直接把这三个对象画出来，这样项目就能从“产品想法”进入“可开工设计”阶段。

## 19. 工程约束补充

为了避免后续架构反复调整，当前项目增加两个硬约束：

1. Python 依赖和环境统一使用 `uv`
2. Agent 编排、工具调用、会话与上下文管理统一基于 Google ADK

这意味着我们后面设计系统时，所有核心模块都要回答一个问题：

- 它在 ADK 里对应什么抽象？

例如：

- 策略解析器更像什么 Agent
- 回测执行更像什么 Tool
- 研究会话如何映射为 Session
- 策略卡片和结果对象如何与 Session State / Artifact 协作
