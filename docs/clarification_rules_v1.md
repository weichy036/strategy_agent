# 澄清规则 V1

## 1. 文档目标

这份文档用于定义第一版产品中的“澄清机制”。

它回答的问题是：

- 当用户提出一个策略研究问题时，系统该如何判断信息是否足够
- 哪些缺失项必须追问
- 哪些缺失项可以自动补默认值
- 如何在“减少打扰用户”和“保证回测可信度”之间取得平衡

这份文档是连接用户流程、Strategy Schema 和 ADK 编排逻辑的关键中间层。

## 2. 为什么澄清机制是核心能力

用户通常不会一次性给出完整、严谨、可执行的策略定义。

例如用户说：

- “MACD 金叉买入，死叉卖出”
- “每个月买入市值最大的 20 只股票”
- “创新高的股票每周买入”

这些说法对于人来说似乎已经很清楚，但对于系统执行来说通常还缺：

- 标的范围
- 频率
- 成交价格
- 调仓规则
- 默认过滤项
- 回测区间

如果系统完全不澄清，结果会失真。
如果系统什么都追问，体验会很差。

因此第一版必须有一套明确的澄清规则。

## 3. 澄清机制的目标

第一版澄清机制需要满足 4 个目标：

1. 尽量少问
2. 关键问题必须问
3. 默认值必须透明
4. 最终一定能收敛到可执行策略对象

## 4. 总体决策原则

系统收到用户问题后，先做一次“字段完整性判断”。

对于每个关键字段，系统只做三种决策：

1. 已明确
2. 可合理推断或默认补全
3. 必须追问

最终目标是把问题转换成一个完整的 `Strategy Schema V1` 对象。

## 5. 第一版字段分类

第一版建议把字段分成三层：

### 第一层：核心语义字段

这些字段决定“用户到底想测什么”。

例如：

- 标的或股票池
- 策略类型
- 买卖条件
- 排序逻辑
- 调仓频率

### 第二层：执行关键字段

这些字段决定“策略会如何被执行”。

例如：

- 成交价格
- 持仓数量
- 权重方式
- 持有周期

### 第三层：默认假设字段

这些字段重要，但第一版允许用系统默认值补齐。

例如：

- 手续费
- 滑点
- 默认过滤项
- 默认回测区间

## 6. 哪些情况必须追问

以下情况在第一版中建议定义为“必须追问”。

## 6.1 标的无法唯一识别

示例：

- “帮我测一下平安银行”
  但系统无法确定是股票、ETF 还是其他同名对象

系统动作：

- 必须追问更具体名称或代码

原因：

- `universe` 无法安全生成

## 6.2 股票池范围不明确且无法合理推断

示例：

- “买入市值最大的 20 只股票”

如果系统不能稳定使用默认市场范围，或者产品当时支持多个股票池入口，就必须追问。

当前第一版建议：

- 如果用户没有明确说明，默认解释为 `A 股全市场`

因此在第一版里，这一项通常不追问，而是默认补全并透明告知。

但如果用户问题语义冲突，例如：

- “买入中国大盘股”

则系统应追问：

- 你是指 A 股全市场中的大盘股，还是某个具体指数成分股？

## 6.3 策略频率无法确定

示例：

- “MACD 金叉买入，死叉卖出”

问题：

- 是日线、周线，还是月线？

系统动作：

- 第一版建议优先默认成日线，并明确提示
- 但如果用户问题中已经表现出跨频语义冲突，必须追问

例如：

- “短线做 MACD 金叉”

这里“短线”不足以确定频率，应该追问。

## 6.4 买卖逻辑不完整

示例：

- “MACD 金叉买入”

问题：

- 没有卖出条件

系统动作：

- 必须追问卖出规则，或者在非常明确的产品模板上下文中提示用户选择默认卖出模板

原因：

- 没有完整退出逻辑，策略含义不完整

## 6.5 排序或筛选方向不明确

示例：

- “买入波动率最大的股票”

问题：

- 是最大还是最小看起来明确
- 但如果用户说“买入低估值股票”，则具体字段不明确：PE、PB、PS 还是股息率？

系统动作：

- 必须追问具体使用哪个字段

原因：

- `selection.ranking.sort_by` 无法稳定确定

## 6.6 持仓数量无法确定且不能安全默认

示例：

- “每个月买入市值最大的股票”

问题：

- 是 1 只、前 10 只、前 20 只，还是所有大市值股票？

系统动作：

- 必须追问

原因：

- 这会显著影响收益和风险结果

## 6.7 持有周期无法确定

示例：

- “筛选创新高的股票后买入”

问题：

- 持有多久卖出？

系统动作：

- 必须追问

原因：

- `selection.hold_period` 不完整

## 6.8 请求超出第一版能力范围

示例：

- 分钟级策略
- 期权策略
- 高频做市
- 杠杆和做空组合

系统动作：

- 不进入普通澄清流程
- 直接说明当前版本不支持，并给出可行替代方案

## 7. 哪些情况可以默认补全

以下情况第一版建议可以由系统自动补全，但必须显式告知用户。

## 7.1 回测区间

如果用户没有给出起止时间，系统可以默认使用一个标准历史区间。

第一版建议：

- 默认起点为当前可用数据的一个固定安全起点
- 默认终点为最新可用交易日

说明：

- 具体日期规则放在 `backtest_assumptions_v1.md`

## 7.2 成交价格假设

对于常见规则型策略，第一版建议默认：

- 信号在当前 bar 形成
- 下一交易日开盘成交

如果用户明确说“收盘买入、收盘卖出”，则覆盖默认值。

## 7.3 手续费和滑点

第一版建议默认使用一套基础成本模型。

例如：

- 手续费 3 bps
- ETF 滑点 5 bps
- 股票组合滑点 10 bps

## 7.4 默认过滤项

对于 A 股组合策略，第一版建议默认过滤：

- ST
- 停牌
- 近 60 日上市新股

如果用户明确要求不过滤，则以用户要求为准，但需做风险提示。

## 7.5 权重方式

对于横截面 top N 组合，如果用户没有说明：

- 第一版默认使用等权

对于单标的策略：

- 默认 `full_position`

## 7.6 市场范围的安全默认

对于用户说“股票”但没有进一步说明时，第一版建议默认：

- `A 股全市场`

但系统必须在策略理解和策略卡中明确展示这个默认值。

## 8. “默认补全”和“必须追问”的分界标准

判断标准建议用下面三条。

### 标准 1：是否会改变用户问题的核心含义

如果会显著改变“测的是什么”，就必须追问。

例如：

- 卖出规则
- 选股字段
- 持仓数量

### 标准 2：是否有行业通用且稳定的默认值

如果存在稳定、透明、可解释的默认值，可以默认补。

例如：

- 手续费
- 滑点
- 等权

### 标准 3：默认补全后是否容易被用户理解和纠正

如果默认值直观、用户看到后容易发现并修改，可以默认补。

例如：

- 回测区间
- 是否使用等权

## 9. 第一版推荐澄清顺序

为了减少打扰用户，澄清顺序建议固定。

### 第一步：先判断是否超范围

如果超范围，直接终止普通澄清流程。

### 第二步：判断能否识别策略主类型

例如：

- 单标的信号策略
- 横截面轮动
- 筛选后持有

如果主类型都无法识别，需要先追问意图。

### 第三步：优先补齐核心语义字段

例如：

- 标的
- 买卖条件
- 排序字段
- 持有周期

### 第四步：再补执行关键字段

例如：

- 持仓数量
- 调仓频率
- 成交价格

### 第五步：最后补默认假设字段

例如：

- 成本
- 过滤项
- 回测区间

## 10. 第一版推荐交互模板

### 模板 A：无需追问，直接透明补全

适用情况：

- 关键字段已齐全
- 只缺可默认项

示例：

```text
我理解你的策略是：
每个月买入 A 股全市场中总市值最大的 20 只股票，等权持有到下个月再调仓。

我将使用以下默认设置：
- 回测区间：默认历史区间
- 成交方式：下一交易日开盘价
- 手续费与滑点：使用系统默认值
- 过滤项：剔除 ST、停牌和近 60 日上市新股
```

### 模板 B：只追问一个关键缺失项

适用情况：

- 只有一个关键字段缺失

示例：

```text
我已经理解了大部分策略逻辑，但还缺一个关键条件：
你这个策略是按日线执行，还是按周线执行？
```

### 模板 C：指出超范围并给替代方案

适用情况：

- 请求超出第一版能力边界

示例：

```text
你这个请求涉及分钟级期权回测，当前第一版暂不支持。
如果你愿意，我可以先帮你把它改写成日频 ETF 或股票策略研究问题。
```

## 11. 与 Strategy Schema 的对应关系

澄清机制最终是为了补齐 Schema 字段。

建议按字段做一层映射：

### 必须追问的高优先级字段

- `strategy_type`
- `universe`
- `signals.buy`
- `signals.sell`
- `selection.ranking.sort_by`
- `selection.hold_period`
- `portfolio.position_count`

### 可默认补全字段

- `period.start`
- `period.end`
- `execution.buy_price`
- `execution.sell_price`
- `costs.commission_bps`
- `costs.slippage_bps`
- `constraints.tradability_filters`
- `portfolio.weight_method`

## 12. 与 ADK 编排的对应关系

这份文档会直接约束顶层 ADK 编排 Agent 的决策逻辑。

顶层 Agent 在接收到用户问题后，应完成以下判断：

1. 是否超出能力边界
2. 是否已经具备构造 Schema 的最低信息
3. 当前缺失字段是“必须追问”还是“可以默认补全”
4. 是否可以进入“生成策略卡”阶段

因此在 ADK 设计里，澄清机制不是附属行为，而是顶层编排的核心职责之一。

## 13. 第一版常见问题示例

### 示例 1：可以直接默认补全

用户问题：

- “对于沪深300ETF，MACD 日线金叉买入、死叉卖出，收益怎么样？”

判断：

- 标的明确
- 频率明确
- 买卖条件明确
- 可默认补的只剩回测区间和成本

结论：

- 不追问，直接透明补全

### 示例 2：必须追问卖出逻辑

用户问题：

- “MACD 金叉买入的效果怎么样？”

判断：

- 买入条件明确
- 卖出条件缺失

结论：

- 必须追问

### 示例 3：必须追问持仓数量

用户问题：

- “每个月买入市值最大的股票”

判断：

- 排序字段初步明确
- 持仓数量缺失

结论：

- 必须追问

### 示例 4：超出范围

用户问题：

- “做一个股指期货 5 分钟均线策略回测”

结论：

- 直接说明当前不支持

## 14. 设计原则总结

### 原则 1：关键语义不能偷补

不能替用户发明核心策略逻辑。

### 原则 2：默认值要为“可信执行”服务

默认值不是为了省事，而是为了把常见问题稳定收敛成可执行对象。

### 原则 3：澄清问题必须短、具体、可回答

不要一次抛出多项复杂金融参数。

### 原则 4：所有默认补全都要可见

系统必须告诉用户自己补了什么。

### 原则 5：最终目标是收敛到可执行 Schema

澄清机制不是聊天，而是结构化收敛过程。
